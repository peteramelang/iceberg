## The inspiration

This project was inspired by the **["Vibe Coding vs Production Reality"](https://www.reddit.com/r/ClaudeAI/comments/1t3bk3x/vibe_coding_vs_production_reality/)** iceberg meme that made the rounds on r/ClaudeAI — a single image that names, in one frame, the gap between "I shipped an MVP this weekend" and "this app survives real users." Above the waterline: Lovable, Bolt, ChatGPT, Cursor, Windsurf. Below it: authentication, payments, billing logic, subscription states, scalability, latency, load balancing, logging, alerting, incident response, disaster recovery, GDPR/CCPA, rate limiting, CI/CD, rollbacks, feature flags, test coverage, instrumentation, retention, churn control, cloud costs, multi-region, idempotency, secrets management, A/B testing, vendor lock-in — and on, and on.

Iceberg is a curated, opinionated curriculum for that submerged half. **46 topics across 7 phases**, each one a hand-picked set of videos, docs, official guides, and courses from the people who actually built these systems — Stripe, AWS, Auth0, Google SRE, OWASP, and 220+ others.

The logo is the iceberg itself, drawn as a graph: a few visible nodes above a dashed waterline, a denser network of dependencies beneath. That's the product — make the underwater part navigable.

## What you get

- **A typed curriculum** — 7 phases (Security & Identity → Money & Data → Observability → Reliability & Scale → Delivery & Operations → Incident Management → Growth & Governance), each with 5–8 topics, each topic with a narrative, curated resources, pitfalls, code examples, and a difficulty rating.
- **A connection graph** — topics aren't a flat list. Every topic is linked to its prerequisites, complements, and follow-ups with typed edges, so you can navigate by dependency instead of by chapter.
- **Learning paths** — pre-baked routes through the graph for common goals (e.g. "ship a SaaS that takes money safely").
- **A web app** — React + Vite, fast, offline-friendly, bookmarks and progress stored locally. No login, no telemetry, no paywall.
- **A reproducible content pipeline** — every topic on the site was generated by an autonomous multi-agent process whose prompts and ledger live in this repo. The content is reviewable, regeneratable, and resumable.

## How the content is built

The curriculum isn't hand-written — it's generated by a deterministic, resumable pipeline executed by Claude Code:

1. **Stage 0** — propose and synthesize the taxonomy
2. **Stage 1** — research each topic with parallel sub-agents
3. **Stage 2** — tiebreak disagreements between researchers
4. **Stage 3** — adversarial challenge + judge (find weak picks)
5. **Stage 4** — liveness check (kill dead links, replace with live ones)
6. **Stage 5** — build the typed connection graph
7. **Stage 6** — enrich (narrative, pitfalls, code examples, difficulty)
8. **Stage 7** — design learning paths

Each stage commits its output, so the run is fully resumable from `content/_ledger.json`. Open the repo in Claude Code and say:

> continue the iceberg pipeline

See [`pipeline/README.md`](./pipeline/README.md) and [`pipeline/runbook.md`](./pipeline/runbook.md) for the full spec.

## Contributing

Iceberg is open source and contributions are welcome. The shape of a useful contribution depends on what you're touching:

### Found a dead link, wrong attribution, or outdated resource?

Open an issue or a PR editing the relevant file in `content/<phase>/<topic>.md`. Resource frontmatter is YAML; keep the schema intact. If you're credited in [CREDITS.md](./CREDITS.md) and want a link changed or removed, open an issue and we'll handle it.

### Want to suggest a new resource?

PRs editing a topic's resource list are the fastest path. Please include:
- a one-line reason it belongs on the list
- the resource type (`video`, `docs`, `course`, `service`, `book`, `tool`)
- the creator name (so they get credited)

### Want to propose a new topic or phase?

Open an issue first. Taxonomy changes ripple through the connection graph and learning paths, so we like to discuss before regenerating.

### Working on the app or pipeline code?

```bash
npm run typecheck
npm run lint
npm run test
npm run build
```

All three must pass before a PR is merged. Keep PRs focused — one concern per PR. The codebase follows the conventions captured in `CLAUDE.md` and `DESIGN.md`; skim those before opening a non-trivial PR.

### Reporting issues

Bug reports are most useful with: browser + OS, the URL you were on, what you expected, what you saw, and a screenshot if visual. For content issues, link the specific topic.
